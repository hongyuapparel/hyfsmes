## 二、订单状态设计

### 2.1 状态列表

请把所有订单状态列出来，并说明含义（下面是示例，可直接改成你的实际状态）：

| 状态 code | 中文名称 | 说明（何时进入、用来干什么） |
|----------|----------|------------------------------|
| `draft` | 草稿 | 新建订单但未提交，允许自由修改和删除 |
| `pending_review` | 待审单 | 业务员「保存并提交」后等待审核 |
| `pending_pattern` | 待纸样 | 审单通过后，等待生产管理纸样管理页面处理 |
| `pending_purchase` | 待采购 | 纸样确认后，等待生产管理采购管理页面处理 |
| `pending_cutting` | 待裁床 | 物料齐备后，等待裁床排产 |
| `pending_sewing` | 待车缝 | 裁床完成后，等待车缝 |
| `pending_finishing` | 待尾部 | 车缝完成后，等待尾部/质检/包装 |
| `completed` | 订单完成 | 所有工序完成并确认 |


> 若你们有自己的一套状态命名，比如「待客户确认」「待打样」「样板确认」等，请按实际罗列。

### 2.2 状态流转规则

用「**从 → 到**」的形式描述每一步状态变化，谁能在什么地方触发，以及需要满足什么条件。

示例表格（请根据实际修改 / 补充行）：

| 当前状态 | 目标状态 | 触发位置（页面 + 按钮） | 谁可以操作（角色） | 前置条件 / 备注 |
|---------|----------|-------------------------|--------------------|-----------------|
| `draft` | `pending_review` | 订单编辑页：「保存并提交」 | 业务员、管理员 | 必填项通过校验；生成下单时间 |
| `pending_review` | `draft` | 订单列表：「退回草稿」 | 审单人、管理员 | 必须填写退回原因 |
| `pending_review` | `pending_pattern` | 订单列表：「审单通过」 | 审单人、管理员 | 审核通过；记录审核时间 |
| `pending_pattern` | `pending_purchase` | 纸样页面：「纸样完成」 | 纸样负责人 | 纸样信息填写完整 |
| … | … | … | … | … |

补充说明（如有）：  
- 哪些状态之间**绝对不允许**直接互跳（例如不能从已完成回到草稿）。  
「自动流转」，上一个状态操作完成后自动跳到下一个工序。

## 三点五、权限配置界面设计（roles.vue）

> 这一节专门约定「权限配置界面」应该如何分组展示订单相关权限，方便后续前端实现。

### 3.5.1 权限分组（仅影响展示，不改后端 code）

在权限配置页的「操作权限」树中，`/orders/list` 下的权限按业务分组展示：

- 草稿相关（group: `orders.draft`）
  - 订单-草稿提交
  - 订单-草稿删除
  - 订单-草稿编辑（新增，code 建议：`orders_edit_draft`）
- 待审单相关（group: `orders.review`）
  - 订单-审核待审单
  - 订单-退回草稿
  - 订单-编辑流程中订单（新增，code 建议：`orders_edit_progress`）
- 生产流转相关（group: `orders.flow`）
  - 订单-纸样完成
  - 订单-采购完成
  - 订单-裁床完成
  - 订单-车缝完成
  - 订单-尾部完成 / 发货 / 入库
- 管理视角（group: `orders.admin`）
  - 订单-全状态可见可编辑（仅管理员勾选）

> 分组只影响前端 UI 展示：在 `/permissions` 接口返回的 `PermissionItem` 上通过 code 或配置映射出 group，再在 `roles.vue` 里按 group 组织为多级树节点。

### 3.5.2 角色权限模板

在「选择角色」右侧增加「应用模板」下拉，用于一键勾选推荐权限组合：

- 模板：业务员
  - 勾选：草稿相关（含 `orders_edit_draft`）、查看列表、查看价格
- 模板：跟单员
  - 勾选：草稿相关 + 待审单相关 + `orders_edit_progress`
- 模板：生产部门
  - 勾选：生产流转相关权限（纸样/采购/裁床/车缝/尾部完成），不勾选编辑草稿/待审单权限
- 模板：管理员
  - 勾选：上述全部

模板只是前端行为：点击后在当前角色的权限树上勾选对应权限 id，仍可手动增删。

### 3.5.3 权限说明（tooltip）

每条与订单相关的权限在配置中需要有清晰说明，前端以 tooltip 形式展示，例如：

- 订单-草稿提交  
  - 作用：允许在草稿状态点击「保存并提交」，将状态从 `draft` 改为 `pending_review`。  
- 订单-编辑流程中订单（`orders_edit_progress`）  
  - 作用：允许在非草稿且非完成状态下编辑订单（例如修改备注）。  

说明文本可以通过硬编码映射（前端根据 `code` 匹配），或后端在 `/permissions` 响应中增加 `description` 字段（推荐）。

- 生产管理-裁床管理/自动显示待裁床状态的订单，生产部门操作裁床登记按钮录入实际裁剪 数量后，流转到待车缝状态
- 生产管理-车缝管理/自动显示待车缝状态的订单，生产部门操作车缝登记按钮录入实际车缝 数量后，流转到待尾部状态
- 生产管理-尾部管理/自动显示待尾部状态的订单，生产部门操作尾部登记按钮录入实际包装 数量，操作发货按钮，财务审核后流转到订单完成状态，或者操作入库按钮，仓库接收后转到订单完成状态
- 库存管理-待入库/自动显示尾部操作入库的订单，接收入库后，订单流转到订单完成状态

## 五、字段与数据规则

这里专门写跟「状态」相关的关键字段规则，避免前后端理解不一致。

- **下单时间（orderDate）**  
  - 定义：第一次从 `draft` 进入非草稿状态时的时间（当前实现为「保存并提交」时写入）。  
  - 展示规则：  
    - 草稿状态订单：列表卡片上**不显示**下单时间。  
    - 其他状态且存在 `orderDate`：显示「下单时间：YYYY-MM-DD HH:mm」。  

- **订单类型（orderType / label）**  
  - 存储字段：数据库 `orders.label`。  
  - 前端字段：编辑页使用 `orderType`，保存时同步到 `label`。  
  - 展示：在订单卡片上用标签形式展示，仅显示值，不显示字段名。  

- **合作方式（collaborationType）**  
  - 存储字段：数据库 `orders.collaboration_type`。  
  - 展示：订单卡片 SKU 行后面用 `· 合作方式` 的形式追加，仅有值时显示。  

- **工艺项目（processItem）**  
  - 仅当编辑页选择了工艺时，列表卡片才以标签形式展示，无字段名。  

如有其它想固化的规则（例如「某状态不能删除」「审核必须填写原因」），也可以写在这里。
