# 尾部管理流程优化建议与方案

本文档针对「登记包装完成」命名、出货财务审批、入库场景与订单结束条件，给出建议与可选实现方案，供确认后分阶段实现。

---

## 一、现状与问题

1. **命名不匹配**：当前按钮为「登记包装完成」，但实际只录入「尾部收货数」和「次品数」，收货 ≠ 包装完成，易产生歧义。
2. **出货无审批**：设计文档要求「操作发货按钮，财务审核后流转到订单完成状态」，当前实现为尾部直接发货即生效，无财务审批环节。
3. **财务审批依据**：财务需在小满中查看对应订单的收款记录，钱收齐后才通过审批允许出货，希望有更省时、方便的方式。
4. **入库场景**：
   - 亚马逊/Shein 等：货物直接入库，后续业务员再找仓库出货（**无需先发货**）。
   - 部分出货 + 部分入库 + 次品：出货一部分、次品一部分、剩余入库，仓管在「待入库」中操作入库。
5. **订单结束**：当「尾部收货数」全部分配完毕（出货 + 入库 + 次品 = 尾部收货数）后，订单才算结束。——**当前逻辑已满足**，无需改。

---

## 二、建议与方案

### 2.1 命名与步骤（建议）

**建议一：拆分为两步，语义清晰**

| 步骤 | 名称建议 | 含义 | 当前对应 |
|------|----------|------|----------|
| 第一步 | **登记尾部收货** | 尾部清点实际收到的数量（可按尺码）、次品数；不表示「包装完成」 | 现有「登记包装完成」弹窗，仅改文案与标题 |
| 第二步 | **包装完成** | 当出货 + 入库 + 次品 = 尾部收货数时，系统自动视为「包装完成」，订单结束 | 现有逻辑已满足，可仅在文案/提示中体现 |

**建议二：若不想改流程，仅改文案**

- 将按钮和弹窗标题改为「**登记尾部收货**」（或「登记尾部收货与次品」），副标题/说明写清：「登记尾部实际清点收到的数量与次品数；后续通过发货/入库将数量分配完毕即视为包装完成。」

**推荐**：采用「建议二」先统一文案，成本低、不改变现有操作步骤；若后续需要「登记收货」与「包装完成」在界面上分开展示，再按建议一拆分。

---

### 2.2 出货 + 财务审批（方案选型）

设计目标：**尾部提交出货申请 → 财务审批（依据小满收款）→ 审批通过后出货才生效**。

#### 方案 A：审批流 + 财务在「财务管理」或「尾部待审批」中操作（不接小满 API）

- **流程**：尾部在「尾部管理」点击「发货」时，不直接扣减/生效，而是生成「出货申请」记录（关联订单、本次出货数、申请时间、申请人）。状态为「待财务审批」。财务在单独页面（如 **财务管理 → 发货审批**，或 **生产管理 → 尾部管理** 的 Tab「待财务审批」）看到待审批列表，每行可展示：订单号、SKU、小满单号、本次出货数、申请时间等；财务**人工**到小满系统查收款后，在系统内点「通过」或「驳回」。
- **通过后**：系统将本次出货数累加到 `tailShippedQty`，与现有 `ship` 逻辑一致；若驳回，可留备注，尾部可修改后重新申请。
- **优点**：实现简单，不依赖小满是否开放订单/收款 API；财务习惯不变（仍到小满查款），只是审批动作在 ERP 内完成，列表集中、可追溯。
- **缺点**：财务仍需切到小满查一次。

#### 方案 B：审批流 + 对接小满订单/收款 API（若小满提供）

- 在审批列表中，由系统拉取并展示「该订单对应小满单号的收款状态」（如：已收齐 / 未收齐 / 部分收款），财务在系统内即可判断，减少切换小满的次数。
- **前提**：小满开放「按订单查收款记录」或「订单收款状态」类 API，且项目愿意做对接与维护。
- **优点**：省时、体验最好。
- **缺点**：依赖小满 API 能力与稳定性。

#### 方案 C：不建审批流，仅靠制度与权限

- 出货仍由尾部直接操作，但「发货」按钮仅对拥有「财务审核发货」权限的角色开放（或仅财务可点「发货」，尾部只能提交「待发货清单」给财务，由财务在尾部页代点发货）。不增加审批表与状态。
- **优点**：开发量最小。
- **缺点**：无审批记录、无「待审批」列表，审计与追溯较弱。

**推荐**：优先做 **方案 A**（审批流 + 财务在系统内通过/驳回，人工去小满查款）；若后续小满有收款相关 API，再叠加 **方案 B** 在审批列表展示收款状态，进一步省时。

---

### 2.3 财务审批入口放在哪里更省时、方便？

- **选项 1**：**财务管理** 下增加菜单「发货审批」（或「出货审批」），列表仅展示「待财务审批」的出货申请，财务只处理审批，不关心尾部其它 Tab。
- **选项 2**：在 **生产管理 → 尾部管理** 中增加 Tab「待财务审批」，与「全部 / 等待发货 / 已发货 / 已入库」并列，财务进入尾部页后切到该 Tab 处理。
- **选项 3**：**首页/工作台** 增加「待审批发货」数量与入口，点击跳转到上述列表。

**推荐**：**选项 1 + 选项 3**：审批列表放在「财务管理 → 发货审批」，并在首页给财务角色展示「待审批发货 N 条」入口，减少找页面的时间。

---

### 2.4 入库逻辑与「直接入库」场景

当前实现要求：**必须先执行发货后才能入库**（`finishing.status !== 'shipped'` 时不可入库）。这与「亚马逊/Shein 直接入库、不发货」矛盾。

**建议**：放宽为「登记尾部收货后即可入库」，即：

- 允许在 **pending_ship**（等待发货）状态下也可操作「入库」；
- 规则仍为：出货 + 入库 + 次品 ≤ 尾部收货数，且三者之和 = 尾部收货数时订单完成；
- 若某订单从未发货（全部为入库+次品），则 `tailShippedQty = 0`，仅 `tailInboundQty + defectQuantity = tailReceivedQty` 时完成。

这样既支持「先发货再入库」，也支持「直接入库、不发货」。

---

### 2.5 订单结束条件（已满足，仅确认）

- **条件**：`tailShippedQty + tailInboundQty + defectQuantity === tailReceivedQty` 时，订单置为已完成，尾部状态置为已入库；若有入库数则写入「待入库」表供仓管操作。
- 当前实现已满足，无需改逻辑；仅需在「直接入库」支持后，对「仅入库、无出货」的订单同样适用上述等式。

## 三、实现阶段建议（遵守项目规则，分步确认）

| 阶段 | 内容 | 涉及范围 |
|------|------|----------|
| **1** | 文案调整：按钮与弹窗「登记包装完成」→「登记尾部收货」，并加简短说明（收货与次品；分配完毕即包装完成） | 仅前端尾部管理页 |
| **2** | 入库放宽：允许在「等待发货」状态下也可操作入库，支持「直接入库、不发货」 | 后端 production-finishing.service（inbound 条件）；前端可选提示文案 |
| **3** | 出货审批流：新增「出货申请」表与状态；尾部「发货」改为提交申请；财务「发货审批」页通过/驳回后生效 | 后端新表、新接口；财务管理下新菜单与新页；尾部管理「发货」改为「申请发货」并走审批 |
| **4**（可选） | 审批列表展示小满单号、跳转链接或（若有小满 API）收款状态 | 前端审批列表列；必要时后端调小满 |

请先确认：  
- 命名是否采用「登记尾部收货」；  
- 出货是否采用「方案 A + 审批入口在财务管理 + 首页待办」；  
- 是否同意放宽入库条件以支持「直接入库」。  

确认后可按阶段 1 → 2 → 3 的顺序实现，阶段 4 视小满 API 情况再定。
