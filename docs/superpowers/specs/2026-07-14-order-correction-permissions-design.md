# 设计文档：订单纠错双权限（强制改状态 + 编辑已提交生产数据）

- 日期：2026-07-14
- 状态：待实现（产品规则已确认）
- 关联：`docs/ORDER_STATUS_FLOW_DESIGN.md`、`docs/PRODUCTION_PERMISSION_AND_RECOVERY.md`

## 1. 背景与目标

生产中常见两类问题：

1. 订单主状态被误点，需要改回或改到正确状态。
2. 订单已完成或某环节已提交，但采购 / 裁床 / 车缝等登记数据填错，需要改正。

经讨论确认：**不做**「撤销完成」「撤回上一状态」；也不以「重置后续工序、重走流程」作为主纠错手段。改为：

- 有权限的人可以**任意改订单主状态**（默认只改状态）。
- 有权限的人可以**编辑各生产环节已提交 / 已完成的登记数据**。
- 状态与数据对不上时，由人对照后分别改；系统不猜测该不该清空历史。

回收站（软删除 / 恢复 / 删除原因）已上线，**本设计不改动**。

## 2. 权限模型（方案 B）

在「设置 → 角色权限」中新增两个**独立**动作权限：

| 权限码 | 名称（建议） | 能力 |
|--------|--------------|------|
| `orders_force_status` | 订单列表-强制改状态 | 将订单主状态改为任意**已启用**状态定义；写操作记录 |
| `production_admin_edit` | 生产管理-编辑已提交数据 | 在各生产模块编辑并保存**已完成 / 已提交**的登记；写操作记录 |

规则：

- 两权限分开勾选，互不隐含。例如只给改状态、不给改生产数据也可以。
- 入口与后端均按 `@RequirePermission` / `authStore.hasPermission` 判断，**不再以 `role code === admin` 作为强制改状态的唯一条件**。
- 种子数据：`admin` 角色继续自动补齐全部权限（含上述两项），避免超管被锁死。
- 建议授权：超级管理员、跟单主管；车间主任等按需只勾 `production_admin_edit`。

## 3. 强制改状态（`orders_force_status`）

### 3.1 入口

- 订单列表：勾选后批量栏按钮「强制改状态」（有权限才显示）。
- 订单编辑页：有权限时显示同入口。
- 回收站中的订单：保持现有行为（先恢复再改状态）。

### 3.2 行为

- 必填：目标状态、操作原因。
- **默认只修改** `orders.status` / `status_time`，并写入：
  - `order_operation_logs`（action 沿用现有 `admin_force_status`，避免历史记录分裂）
  - `order_status_history`
- **不自动**：回滚辅料出库、回滚成品入库、清空各生产子表明细。
- 「重置后续工序完成标记」：**P0 隐藏该勾选**（接口可暂留参数但不从 UI 暴露；默认 `resetSubsequent=false`）。纠错以手改生产数据为主。若后续确实需要，再在 P2 以折叠「高级选项」恢复，且默认关闭。
- 若目标状态对应环节的子记录仍为「完成」，弹窗**强提示**：定时调和约 2 分钟内可能再次推进主状态，请同步使用 `production_admin_edit` 修正子环节，或再次调整状态。

### 3.3 与现状差异

- 现状：仅超级管理员 + `AdminRoleGuard`。
- 本设计：改为 `orders_force_status` 权限守卫。

## 4. 编辑已提交生产数据（`production_admin_edit`）

### 4.1 覆盖模块

以下模块在登记「已完成 / 已提交」后，对持有 `production_admin_edit` 的用户开放编辑与保存（至少含列表抽屉 / 完成弹窗的更新路径）：

- 采购管理（采购 / 领料登记）
- 纸样管理
- 工艺管理
- 裁床管理
- 车缝管理
- 尾部管理

无该权限时：保持现有锁定（完成后不可改或仅只读）。

### 4.2 行为约束

- 保存时写操作记录：订单号、模块、操作人、时间；尽量带变更摘要或原因（若某模块已有备注/原因字段则复用，否则最少记「纠错编辑」）。
- **不**因纠错编辑自动改变订单主状态（主状态只由正常流转或 `orders_force_status` 修改）。
- 库存联动：沿用各模块现有规则；本设计不新增「编辑即自动还库」。审单自动出库的辅料仍人工核对。

### 4.3 实现注意

- 各 Mutation 接口需在「已完成仍允许更新」分支校验 `production_admin_edit`（或等价服务内断言），避免仅前端放开。
- 前端按钮 / 抽屉「编辑」对无权限用户不可见或禁用。

## 5. 明确不做

| 项 | 说明 |
|----|------|
| 生产岗「撤销完成」 | 不做 |
| 跟单「撤回上一状态」独立能力 | 不做（需要时用强制改状态） |
| 改状态时默认重置子工序 | 不做为默认行为 |
| 改状态自动还原辅料 / 成品库存 | 不做 |
| 改回收站规则 | 不在本范围 |

## 6. 明确保留

- 回收站：软删除、恢复、`orders_restore`、删除原因选项。
- 正常生产推进：各岗位完成动作与流程配置、定时调和。
- 超管编辑订单内容时跳过流程自动重算（既有行为）；与本权限正交。

## 7. 风险与对策

| 风险 | 对策 |
|------|------|
| 只改状态、子工序仍完成 → 被自动推回 | 弹窗强提示；靠 `production_admin_edit` 改子数据；高级「重置完成标记」可选 |
| 更多人可改历史数据 | 双权限分离 + 操作记录审计 |
| 状态与库存不一致 | 文档与 UI 明确不自动还库，人工处理 |

- 前端强制改状态入口：从 `isSuperAdmin` 改为 `hasPermission('orders_force_status')`。

## 9. 验收要点

1. 角色只勾 `orders_force_status`：能改状态，不能改已完成裁床等数据。
2. 角色只勾 `production_admin_edit`：能改已完成生产登记，列表看不到 / 调不通强制改状态。
3. 两权都勾：能改状态且能改各环已提交数据；操作记录可查。
4. 无权限用户行为与改造前一致。
5. 回收站删除 / 恢复 / 原因选择仍可用。
6. 强制改状态默认不清理生产子表；辅料出库不自动回滚。

## 10. 实现分期建议

1. **P0**：权限种子 + 强制改状态改为 `orders_force_status` + 默认不重置 + 冲突强提示。
2. **P1**：各生产模块 `production_admin_edit` 放开已完成编辑（建议按裁床 / 车缝 → 采购 → 其余推进）。
3. **P2**：文档与前端权限文案整理；视需要恢复「重置完成标记」为折叠高级选项。
