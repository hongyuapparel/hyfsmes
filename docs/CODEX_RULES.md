# CODEX RULES

这份文档用于约束 Codex 在本项目中的工作方式。
目标是：在不破坏现有方案的前提下，稳定、谨慎、可追溯地协助开发。

## 1. 默认工作原则

- 只修改用户明确指定的页面、模块、文件或问题范围。
- 不因为“顺手”或“看起来更合理”而扩散修改到其他模块。
- 如果为了完成当前需求，必须改动用户未点名的文件，必须先说明原因、影响范围，并等待用户确认。
- 优先做根因修复，不做临时补丁式修复。
- 优先使用框架、组件库、官方 API 和现有项目模式，不绕过现有体系硬改。

## 2. 开始任何任务前的读取顺序

当任务涉及业务逻辑、字段配置、订单流程、页面结构设计或现有方案延续时，Codex 先读取相关文档再动手。

基础必读：

- `docs/PROJECT_CONTEXT.md`
- `.cursor/rules/project-context.mdc`
- `.cursor/rules/development-principles.mdc`
- `.cursor/rules/project-tech-stack.mdc`

按任务补充读取：

- 涉及字段配置、字段显示、字段顺序、表单/表格字段复用：
  - `docs/FIELD_CONFIG_DESIGN.md`
  - `.cursor/rules/field-definitions.mdc`
- 涉及订单、订单状态、订单编辑页、生产相关流程：
  - `docs/ORDER_STATUS_FLOW_DESIGN.md`

如果用户在对话中额外指定了文档或设计说明，优先遵守那些文档，不自行发明新规则。

## 3. 技术栈与实现偏好

本项目默认技术栈：

- 前端：Vue 3 + TypeScript + Vite + Pinia + Vue Router + Element Plus + Axios
- 后端：NestJS + TypeScript + TypeORM + MySQL

实现时遵守：

- 前端优先使用 Composition API 和项目现有模式。
- 后端接口优先沿用 DTO、校验器、现有模块结构。
- 样式优先使用 Element Plus 官方能力、组件属性、主题变量。
- 非必要不使用 `!important`、强制覆盖、深度选择器去压组件默认行为。
- 新逻辑优先复用现有工具、字段定义、接口封装和公共组件。

## 4. 字段系统规则

如果任务涉及“同一个业务字段在多个页面展示/编辑/筛选”的情况，必须先检查现有字段定义和已有页面做法。

遵守以下原则：

- 同一业务字段在全系统使用统一 `code`、统一含义、统一展示方式。
- 表单、表格、筛选、排序优先基于 `fields/*` 配置，而不是页面内硬编码。
- 已经约定展示形式的复用字段，在其他页面继续沿用相同形式，不随意换组件或换数据源。
- 禁止同一字段在不同页面出现不同命名、不同 code、不同表现而没有明确设计依据。

## 5. 页面设计与需求不清时的处理方式

如果用户只描述业务需求、不会写代码，Codex 应先帮助澄清并给方案，再决定是否直接实现。

优先流程：

1. 先理解页面目标、使用角色、核心操作和信息优先级。
2. 给出 1 到 3 个可执行方案，并说明优缺点。
3. 在没有明显设计结论前，不盲目大面积写代码。
4. 用户选定方向后，再进入实现。

如果用户明确要求“你直接设计并实现”，则可以先给简要方案，再直接落地。

## 6. 文档同步规则

如果本次改动属于需求级别变更，例如：

- 新增模块
- 调整关键业务流程
- 修改订单状态规则
- 统一字段配置方案
- 重构某个核心页面

则在完成代码后，轻量同步文档：

- 在 `docs/PROJECT_CONTEXT.md` 中补一条简短记录。
- 如有必要，再补充对应设计文档。

如果只是小样式调整、局部小 bug 修复、非业务性微调，可以不更新文档。

## 7. 回复用户时的说明方式

完成较大修改后，Codex 回复中应简要说明：

- 本次主要改动了什么模块或页面
- 涉及了哪些关键文件
- 是否同步更新了相关文档
- 是否存在未验证项、风险点或需要用户确认的点

如果没能运行测试或没法完整验证，也要明确说明。

## 8. 安全边界

- 不主动重构未被要求的模块。
- 不主动改动数据库结构、接口契约、全局样式、公共基础组件，除非用户明确授权，或为了完成任务必须修改并已获得确认。
- 不覆盖、不回退、不清理用户已有改动，除非用户明确要求。
- 发现现有代码、文档或规则互相冲突时，先停下来说明冲突点，再继续。

## 9. 用户可直接使用的指令

以后用户可以直接这样对 Codex 下指令：

“遵守 CODEX RULES，先读取相关规则和文档；只改我指定的范围，如果要改其他文件先告诉我。”

如果任务涉及页面设计，也可以这样说：

“遵守 CODEX RULES，先给我页面方案和建议，我确认后再实现。”

## 10. 规则优先级

优先级从高到低：

1. 用户当前这次对话里的明确要求
2. 用户当前指定阅读的设计文档或说明文档
3. 本文件 `docs/CODEX_RULES.md`
4. 项目现有 `.cursor/rules/*.mdc`
5. 项目当前代码中的既有实现模式

如果低优先级规则和高优先级规则冲突，按高优先级执行，并在回复中说明。
